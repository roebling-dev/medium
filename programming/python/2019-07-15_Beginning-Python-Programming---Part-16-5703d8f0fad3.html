Testing

The two most popular modules for testing code are unittest and PyTest. I’m choosing to cover unittest in this article because it ships with Python.

unittest is a framework provided by Python that allows you to create assertions about your code. If the assertions are true, the test passes.

Before we get into testing, we should cover a few things about testing code, some which the community is very divided over.

There are several different forms of testing:

  • Unit Tests — Tests that cover individual parts of your code (what this article is about)
  • Reliability Tests (smoke tests) — Tests that ensure your full program will withstand the test of time or doesn’t crash on you. (The name smoke test derives from an old knack that if you turn something on and it doesn’t start smoking, it should work.)
  • Quality Assurance Tests — Tests performed by the QA team if you have one. QA engineers are trained to be chaos monkeys to your code. (To the QA engineers: I say this with love.)
  • User Acceptance Tests — Your code goes to a pilot group who tests your code. The larger the pilot group, the more results you’ll get back. (e.g., Early Access games)

These tests can go in any order, but this is the typical order of use.

  • Code Coverage — The percentage of your code that is covered by tests.
  • Coverage Ratio — A ratio that defines the number of tests to the number of lines in your code base.

I know these are touchy subjects so I’m going to discuss it rationally.

100% code coverage is always great to have, but at the very least you should focus on the business logic of your app, the main part of the code that your program was created to do. In most cases, a good bare minimum should be 30% coverage, although you can easily get 80% coverage with a few well-designed tests. A well-designed test is one that covers a wide workflow of your code, such as checking the response from a web server app to ensure it is correct. You get the route, the business logic, and even the database logic all included in a single test.

Coverage ratios are usually a bragging point for some devs. Some will say “I have a 1:1 ratio” (one test for one line of code). This probably means a lot of wasted cycles of productivity. A better goal is 1:10 or even 1:20 in medium to large programs. This provides a middle ground between wasted productivity and testing accuracy.

  • Test-Driven Development (TDD) — TDD is the method that contrasts with code first, then write tests. TDD is writing a test first, then writing the minimal amount of code to make the test pass.

TDD and Coverage ratios usually go hand in hand; if you ever hear a developer talk about their test ratios, they are probably using TDD. TDD is not a bad thing, but it can quickly cause maintenance headaches if you are not careful.

Time for an example.

First, we need to import unittest to pull in the module for testing. Next, we have a basic add function that adds two numbers.

Now we create a class named TestAddFunction, which inherits from unittest.TestCase and will contain every test we’ll create for the add function above. In a real-world scenario, your class will contain all tests related to the functionality of a workflow. If the workflow is retrieving names from a file, you’d test opening a file, reading the file, parsing the names, and ensuring the values and number of names retrieved from the file are correct.

The first test we create is test_add_numbers. Its sole responsibility is to test that our add function returned the correct result. First, we call the function and store the result in a variable. Finally, we use 
self.assertEqual(result, 4, "Whoops") to check that the result of our function was indeed four. Since we inherited from unittest.TestCase, we receive quite a few assertions to help us test our code.

assertEqual takes three parameters. The first two are the values we want to compare, and the third is the message we want to print to the console if the test fails. In this basic example, it’s obvious why the test failed, but sometimes you might have more than one assertion, and this helps to determine which assertion failed.

In order to run tests, you can use the following commands:

cd /path/to/project
python -m unittest

If you created a virtual environment in your project directory, you would need to specify a filename at the end:

python -m unittest tests.py

In PyCharm, you’ll need to add a run configuration for tests.

SetUp and tearDown

Sometimes you need to perform work before and after your tests. setUp and tearDown allow you to configure your test environment, so you do not test in a production system.

This is just a simple test that ensures a file was not empty. Here we use self.assertTrue(len(contents), "Empty file") for our test. assertTrue requires a boolean expression. Empty strings, None, and 0 also equate to False results.

Our main focus is setUp and tearDown. setUp is called when our class is created. Think of this as def __init__(self): for test classes. Here we use it to create a new file with the word “Hello” inside. As the test continues, it will call test_file_read, which will call the function read_file and store the response into contents. We then check to make sure contents wasn’t just an empty string; if it was, we print “Empty file.”

Our final check is to assert that the file’s contents are equal to “Hello.” This may seem like a duplicate test, but by first checking the length of the contents, we can be assured the file was at least present. The final test is to ensure the file’s content was as we expected.

When all of our tests run for this class, tearDown is called. This is where we perform any cleanup needed. In this scenario, we created a file named testfile in our project directory, and we need to remove it.

Coverage

There is a lot more to cover on testing, but I’ll leave that to you in the suggested reading below. I want to talk about coverage because it’s how you know when you’re done writing tests. Without coverage, you’d need to follow the adage “Write tests until the fear of failure becomes boredom from writing tests.”

You can install coverage by using pip install coverage or by using the bundled coverage in Pycharm. (Settings > Build, Execution, Deployment > Coverage > Use bundled coverage.py)

You can run and check test coverage from the command line by using the following commands:

cd /path/to/project
python -m coverage run
python -m coverage report

This will print output similar to this:

Name      Stmts   Miss  Cover
-----------------------------
tests.py     13      0   100%

Name is the file that contains tests, Stmts is the number of Statements that were found in the file, Miss is the number of tests that were missed by coverage, and Cover is the percentage of code that is covered by tests.

To see a list of options, use python -m coverage help.

In PyCharm, choose Run > Run tests with coverage. Lines shaded green are covered by tests. You will also see the statistics for each file in the right sidebar (default layout).

Test-Driven Development (TDD)

TDD is a methodology for testing code. The idea is that you’d write a failing test, then write just enough code so that the test passes. The most straightforward way to describe this is to follow the following steps:

  1. Add a test
  2. Run the test. If it passes, go to step one.
  3. Make a change.
  4. Run the test. If it passes, go to step one. If it fails, go to step three.

In our previous examples, we wrote a function, then we wrote the tests that would invoke the function in one go. In TDD, we’d write the test and run it, fully knowing that the test will fail because the logic doesn’t exist. Then we’d write just enough code to make the test pass. If it still fails, we go back and check our logic to figure out why it failed and make corrections. If it passes, we move on.

Since this is just a methodology, I will not cover it here, but I will provide a link in suggested reading if you want to learn more.

Summary

We learned how to make some basic tests in Python and use code coverage to figure out how much of our code is covered by tests, and had a brief introduction to TDD.

I used to have difficulty desiring to write tests because most programs I wrote were simple. I had to change my mindset. I started thinking that each program I wrote would earn $1M a day until it failed. To keep it from failing I needed to write tests that ensured it would run for decades without maintenance.
