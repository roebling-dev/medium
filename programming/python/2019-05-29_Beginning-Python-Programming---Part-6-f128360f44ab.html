It gave us a good understanding of how to break up repeatable code into functions and how the scope of our variables might affect our program’s output.

Today, we are going to learn about a new type of object which can also be considered a type. This new type of object can be considered a container or a blueprint that defines a special scope called a namespace, for similar pieces of code.

This new object is called a class, and it does wonders for cleaning up and organizing your code. As we move through this tutorial, I want you to think of a blueprint for a house, or maybe the cookie cutter that made all of those stars in the image above. It’s not the house, nor the cookie, but it ensures every instance of the blueprint is exactly the same—which is just what a class does.

Classes

Simply stated, classes are objects that contain variables and functions, but in reality, they do so much more than just acting as a container. Classes provide a way to create reusable code that can be used over and over again.

I’m going to go back and reference pointers (there is a reason we covered it, you know). When you create a class you create a reference; that is, point to a block in memory that will store the class, and reference back to it to create new instances, or copies, of this class and copy them to new blocks of memory with new values. This might sound confusing, so maybe some examples will help:

We define a class by using the syntax class <name>(). Classes are named using PascalCase, which means no spaces and the first letter of each word is capitalized. For this example, I’m creating a new House class. The parentheses are required and I’ll explain why further down. As always, we need to use a colon at the end to include the body of the class.

You’ll see something interesting here: def __init__(self). At the very least, this is what your initializer function should look like. Seasoned programmers like to call this the constructor method.

A method and function are mostly interchangeable when used in a sentence. Most programmers won’t correct you if you use it wrong, but the difference between functions and methods is that methods belong to class objects, while functions do not.

The double underscore or __ on both sides of init is referred to as a dunder and is used to overload, or replace, the functionality in a default method. Classes by default contain an __init__(self) method, so if you don’t have anything to do up front, you don’t have to add this method.

But I didn’t use just self in the constructor method, I have a few other parameters: number_of_windows and number_of_doors. I think it’s pretty easy to understand what these are for because we are getting better at naming variables, but why do we need to add them?

In a constructor method, it’s always useful to add parameters to allow us to provide initial values for our class. In this case, we add the number of windows and doors. But that alone doesn’t do the trick. If you just pass parameters to a regular function and don’t do anything with them, nothing happens, right!? We need to assign these to the class, but there’s this issue with scope.

I told you in the last post that Python handles scope for you. This is one of those examples where it does just that. We need to assign the values passed to the constructor method to variables that belong to this instance of the class. For this, we use the keyword self. self means this object, not the class but the object to be created from the class.

self.number_of_windows is the property that we will use to store the number we pass in for the number_of_windows parameter. The dot . is important here. If self refers to this instance of the class, and self.number_of_windows is the property in this instance where we want to store this value, this must mean that . must mean that the property is a sub-object of this class. The . is how we access properties and methods of classes.

Properties are much like methods, except they refer to variables within classes. Yes, they are sometimes used interchangeably and you most likely won’t get corrected.

When creating a new class, you’ll notice I have the following lines:

my_house = House(number_of_windows=2, number_of_doors=1)
your_house = House(4, 2)

Both of these are the correct syntax, except one uses named parameters. If you missed my article on functions, I cover this a little, but the same applies here. If you use named parameters, it’s easy for everyone to understand what the values mean without digging through code. You also get the added benefit of using them in any order you want. In addition, note that self is missing, we do not use self when calling methods. More on self in a minute.

If we choose to omit the named parameters as we do for your_house, we must use the same order for the parameters as they appear in the constructor method. This means that the number of windows is 4 and the number of doors is 2.

Finally, we print the number of windows and doors for each house. my_house only has 1 door so it prints the number 1. your_house is better than mine so it prints 4 windows. I could have printed your_house.number_of_doors and it would have displayed 2. The reason for this is that my_house and your_house are two different instances of House. Even though they were derived from the same class, they are two separate entities.

So why is self so important? We haven’t covered static methods yet, but we can quickly go over them. Static methods are methods that are not dependent on class properties or methods to function. Let’s look at an example.

Here we have a Dog class. Skipping past the constructor, we have our first method. It’s a class method so we include self. We do this because it references back to the name and age property of this instance. However, something all dogs are born with is the ability to speak. When we refer to dogs we know that they should go Woof! Since we know this is a fact, we could make this a static method by adding a decorator above this method named staticmethod. The @ denotes that this should decorate the next function and yes, it is required.

By using the @staticmethod decorator we do not have to use self. So what’s so important about self? If we use self, we have to have an instance of a class. If we don’t have an instance of a class, we cannot use that property or that method that belongs to this class. By using a static method, we are able to call this method without creating an instance of a class.

This means we can use Dog.speak() and we will print out Woof! to our terminal. Datetime is a good example that uses static methods.

from datetime import datetime
datetime.now()

Don’t worry too much about the first line right now—we’ll cover that in the future. Here you can see the datetime class has a static method called now() that returns the current date time from your system. I was able to call the now method without actually having to create the datetime class.

If your method is not marked as static, you will need to create a new instance of the class before you can use it.

Something to note about the get_description(self) method in the Dog class: if you call this method from within the class, you must use self.get_description(). Just because it is a method of this class it doesn’t mean it is exempt from self. The static method speak() must be called with self when calling it from another method within the class.

Properties

No, not those kinds of properties, but I did choose that image for an example of a cookie-cutter blueprint.

Scope matters when you add properties to classes. Time for an example:

What’s the difference? In Person1 we add a property called name, then we create fred and wilma from Person1. We have two instances and we set each to their respective names.

When we display fred.name it incorrectly prints “wilma,” but when we display wilma.name it correctly prints “wilma”. What gives?!

When you create a top-level property without using self., as we do in the constructor method, it becomes a class attribute.

The difference between a class attribute and a class property is that a class attribute is shared across all instances of the class, while a class property is only available for this instance.

Scope got us here, but if we were to change name to self.name our code would be correct; however, we wouldn’t have a constructor method to provide an initial value for self.name upon creation of the instance.

In our second example, we do not use self.name as a property in the class, however, we assign the parameter name in the constructor to self.name. Behind the scenes, when you use self.something, Python automatically raises the scope to the top-level of the class and marks the object as a property rather than an attribute.

class Dog(object):
    self.name
    def __init__(self, name):
        self.name = name
# is the same as
class Dog():
    def __init__(self, name):
        self.name = name

So with that, be careful about how you create your classes and always try to use a constructor.
