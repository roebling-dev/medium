Type Casting

I didn’t just randomly pick a caterpillar picture because it looked nice, ok maybe I did. It actually reminded me of typecasting. Not that your variables go into some shell and come out like butterflies, but because as long as it makes sense, your variables can be cast as another type.

In layman’s terms, typecasting is the process of converting a variable from one type to another. You can convert almost anything to a String type, it won’t always make sense when you print out the value, but it’s possible. Let’s see how you do this, then I’ll come back around and explain what does not work.

Typecasting can also be done with Integers and Doubles and Floats:

When you typecast integers, floats, and doubles this way, the result will become optional by default. The reason for this is because what if the cast didn’t work? We’d need to return nil. This ensures you always have something to work with even if you declared the variable at the same time you performed the cast.

You can even perform type casting on Classes and Structs but in order to do that, we need to understand the other way to typecast objects.

I’m going to split that new syntax up a bit and relay it back to you as the program performs each action.

In the first part, integerValue as means get ready to cast this variable to a different type, the ? means this will be an optional cast, if we succeed give me integerValue as whatever optional type I specify next, otherwise just give me nil. And finally, we have String this is the type we are casting to. Since integers can become strings, if we printed the value of stringValue we’d get Optional(42). Now I’ve shown you this before, this is exactly what is printed out, Optional() just tells you the value inside of the parenthesis is what is currently there but it could be gone in a while. Do not try to use this directly unless whatever function you are calling takes an optional type for the parameter. There are plenty that do.

So we can cast all day long using this syntax and I’m going to give you a few more examples for you to figure out on your own, if you’ve been following along this shouldn’t be a problem for you.

So I threw a couple of curveballs in there, these are all legitimate. While some may not be practical, it gives you some good examples of how things might get casted to different types throughout your code as they are needed in different formats.

In type casting classes, sometimes you need more than what the default implementation of a class gives you. One example that directly relates to iOS programming is UITableViewCells. The default Table View Cell (think of a contact in your phone before you tap it to see the details) only contains a title label and a subtitle if you choose this type of table view cell. However, if you look at your contacts app, it also has a placeholder for an image.

Ok, so if this is confusing, it’s ok, sometimes code can be confusing but I tried to keep it as simple as possible for the example. So first let’s talk about that CustomTableViewCell. CustomTableViewCell is a class that we created and it subclassed from UITableViewCell which is what Apple created. UITableViewCell contains all of the logic we need to display those table view cells on the screen. We just adopt everything that Apple already wrote then gave it properties that we can hook up to our views. Since Apple doesn’t give us a UIImageView, or basically a placeholder we can display images to users. We went ahead and added that to our class but with a weak ownership since the view may disappear before our CustomTableViewCell class is deinitialized. I know that we force unwrap the image and labels using! but Apple does that for us because it knows the label or image will have a value before we use it.

The @IBOutlet thing isn’t really important for now, but in case it’s bugging you to death, @IBOutlet is just used for Interface Builder, the “at” sign just tells IB that it can use this for the interface, and Outlet describes that this is an outlet for the code to work with the view. There is a sibling called @IBAction which is used to hook up actions users can take with your code so we can handle user interaction.

OK, hopefully, I didn’t lose anyone with that, if so, don’t worry, it will all make sense in time.

For the second half, of that example, we layout re-useable views but the default reusable view is of type UITableViewCell. We don’t want re-usable views of UITableViewCell because it doesn’t contain all of our properties that we need to display the cell as we really want it to look. So we type case the cell as? CustomTableViewCell. This allows us later in the code to set values for the current cell which will now appear as CustomTableViewCell along with all of its properties.

We do something called downcasting whenever we perform a function or use a property from a superclass. For instance, in the example above, if I were to call cell.backgroundView. I’d be referencing a property of the superclass because backgroundView was never mentioned in theCustomTableViewCell class.

Take a deep breath, we are done with typecasting and I know that last part was a bit tricky because we haven’t talked about most of that, but don’t worry, this is just the fundamentals. You need to understand the concepts behind these things before you go off working with table views and cells.

There are also two more types that you should know at this point, Any and AnyObject. The difference between these two is simple. Any can represent any value except for function types. AnyObject can represent any instance of a Class (String, Int, Float, Double, class, etc).

When you use Any and AnyObject, you tend to do type casting a lot, I’m talking 99% of the time. There are use cases but when you come across them you will know it, these types are meant more for sending messages to other objects in your code which is why I didn’t cover them in the Types section.

Safely Unwrapping Optionals

OK, this is one of those fun parts I keep telling you about. It’s probably not as fun as baseball if you like playing sports or drinking with friends. But when it comes to programming, we want our code to be SAFE!. By this I mean, we do not want to be using nil values. We want to make sure that at every step where we need a value we didn’t create by hand, we either have a value, or we handle it appropriately.

Now a bit of a disclaimer, there may be a debate out there in the programming world that I don’t know about regarding the following. To be honest, this works great for me, it’s easy to understand, and I believe people have a right to make up their mind on what way is the right way. If this leads to a discussion in the comments, so be it.

There are a few ways that we can unwrap optionals, the first of which is more common in Swift code.

  • if let
  • if var
  • guard let

Let’s go with most common to least common. if let statements are used to assign values to constants for use inside of if statements.

Using if let allows us to remove the optionality so we can use the value as a real value. In the example above, possibleInteger is an optional Int provided earlier in code, if we can successfully cast possibleInteger to an Int then 
if let takes the value and assigns it to actualInteger.

This is the equivalent of saying let actualInteger = possibleInteger as! Int, except actualInteger is only available in the scope of the if let statement.

Here’s the caveat, if we cannot cast possibleInteger to an Int type, the if statement evaluates to false and the entire code block is skipped.

if let statements can also be used just to unwrap already optional values without changing type such as this:

It’s about like saying if this code statement works, then do something here with the value we retrieved from the statement.

But Bob, what if I need that value and don’t want it nested in an if statement?

Well, the people at Apple have a solution for us and that’s the guard statement.

guard acts as a bodyguard for your functions. If the crowd is right, allow the party. If things are not as they are supposed to be, shut the party down.

Here we have an add function that takes two optional Ints for its parameters. The first thing we do is check that both parameters have values. If they do, we know we are good to proceed. If they do not, we exit early out of the function with a default value.

So why not use the guard statement all of the time? Well because sometimes there is other logic involved that should run regardless of whatever values were passed in.

As an example, I was working on a networking piece where sometimes we’d have HTML headers to send in our request, sometimes we wouldn’t. The headers were passed into a function that would perform the request as an optional parameter. Then in the body of the function, I could use an if let statement to read the header parameters that was passed in. If we returned a value, I knew I could loop through the headers and add them to our request. On the other hand, if they didn’t exist or we didn’t need them for this request, it wasn’t a problem, we just skipped over the part adding the headers to the request then started the request as it was without headers.

In short, use if let when you temporarily need the value unwrapped, or the value is not required for you to proceed. Use guard when you must have the value to proceed.

One last thing about if let and then we will move on. if let works just like a regular if statement. You can have an else if or an else clause. So just because you might fail the initial if let doesn’t mean you can’t try again using else if let.

Finally, if var. I think you might already know what this does, but since we’ve already covered if let it’s super easy. if var is exactly the same as if let except it creates a variable, not a constant, for you to use in the body of the if var statement. You can even mix the two up in the same if statement.

And yes, you can unwrap multiple variables and constants by just using a comma after each evaluation. You can even use it on the same value because the variable is declared and instantiated by the time you hit the comma.

Access Control

So let’s talk about the visibility of your code and how it relates to the rest of the code world. Whenever you create a new project, you are creating a module. When you adopt someone else’s code as a framework (a module that you can use to extend your own app functionality), you are adding another module to your code. By default, your code cannot be used by their module, you have to poke holes in your protective shell to allow another module to access your code.

The order of how your code is encapsulated from least restrictive to most restrictive is as follows:

  • Everyone
  • Module
  • File

Don’t scream at me all of you seasoned developers, I can explain.

There are five different types of access that you can provide to classes, structs, and enums. Let’s go over those now, then talk about them in detail in the order that makes the most sense.

  1. Open Access
  2. Public Access
  3. Internal Access — Default access level when not specified
  4. File-Private
  5. Private

Internal Access

Internal access is the default access level when you don’t specify an access control. While it is not required to type internal before the class declaration, in larger projects it is helpful to have internal before the class declaration for readability.

Internal access allows classes, structs, and enums to be used anywhere in the module they were declared in. This means it’s global to your code only. If you add another module to your project, even if you created it, the new module will not be able to access anything with an internal access. You would use this for building the internal structure of your framework.
Simply stated, use this for the things you need to do your work that no one else would ever need to use.

Private Access

private access only allows you to use an entity (class, struct, or enum) where it is declared. If you declare a method inside of an entity as private, the function can only be called by the entity itself, you cannot use this from another entity. If you have an extension (more on this later), of an entity, it can only use private methods of the entity it is extending if it resides in the same file (more on this when we talk about extensions).

File-Private Access

fileprivate is used when we want an entity to be available for other entities that reside in the same file. This is different from private access, in that other entities inside of the same file as the entity defined as fileprivate can access its methods.

Public Access

public access is used when you want to allow this entity to be accessible by other modules in your project. Think of this as an API for your code, you don’t tell everyone how you do what you do, but you do tell them this is how you can get what you need.

Open Access

I’ll be honest, I only learned about this while I was writing this for you. That’s how much I used it. open access is almost the same as public access except open access allows other modules to subclass from or override methods in the class defined with Open Access. Basically, you are allowing someone to use your car and hot-rod it with this kind of access. This is about like saying do whatever you want, where ever you want. Be very careful with this or you could expose your logic to other developers that you may not want (trade secrets).

So let’s recap this using my car as an example in relation to code.

Private access should be reserved for things that would function much like a piston, A piston takes in a mixture of air and gas (parameters) compresses it (logic), and ignites it (more logic) before creating power (return type). Bonus: if either of those are nil, the car stops (crash), guess we should have safely unwrapped them (ba-da-dum-tis, stupid I know).

File-Private would be like the fuel line, it goes from the gas tank (function to read from a database) to the engine and pumps into the piston (function to insert gas), both are defined within the same file and work well together. They don’t need any extra help and all is good.

Internal access would be the equivalent of everything that happens under the hood of the car, I don’t really care how my car gets me from point a to point b, I just want to get there.

Public access is using the key in the ignition to start the car, using the pedals to adjust speed, and the steering wheel to steer. I don’t really care how the car goes, I just know that if I call the function stepOnTheGas(with: pressure) my car moves, likewise if I call the function stepOnTheBreak(with: pressure) my car stops.

Open access is about the equivalent of ripping out the radio so I can put an aftermarket stereo in. Or maybe adding a turbo to the engine if I change its access to be less restrictive.

In case you were thinking crazy and were curious about adding a nitrous booster to the car, the engine would be open access, but the nitrous itself would just be public access because we don’t care how the nitrous works, we just know if we push the little red button, we introduce a new element into the piston method and that’s it.

OK, so enough about access controls let’s wrap this thing up.
