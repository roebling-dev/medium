Error Handling

Have you ever gone into the kitchen to prepare food and realized there are no clean dishes?

You have a few choices:
A. Abandon Dinner
B. Go out to a restaurant, or
C. Clean the dishes, then fix dinner at home

These decisions not only affect you but also the people you are feeding.

Option A is the easiest solution, but everyone starves and looks at you to be the savior of dinner.

Option B is also easy but it is more expensive, takes a little extra work you wouldn’t have had to do, and doesn’t always taste as good as a home-cooked meal.

Option C is the difficult path but it is the best solution if you want your food to taste how you want it with few complaints, and hopefully no broken dishes.

So what does all of this have to do with error handling? Everything!

When you decided to try to fix dinner, you called a method.

We really don’t think too much about error handling. If this was our app, maybe half of our users would actually get food, the rest would break a plate and get nothing.

We do this so naturally on a daily basis that when we get into programming, some developers just let their apps crash and say “oh it’s the user’s fault.”

[Buzzer Sounds] Wrong answer! If an app crashes while someone else is using it, it is YOUR fault!

It is always your fault because you didn’t handle it. This is what QA teams are for in businesses. To make sure that your user can’t be that guy or gal.

I want you to think for a minute, high level, about where the places something could go wrong in fixDinner() above. Really.

So I hope you actually did that instead of continuing to read on. But the answer is five main things could go wrong in that code above, and each can have multiple things wrong inside of it. Without more information the answer is ambiguous. But I also said high level, so this means five things.

  1. You do not receive any dishes.
  2. You do not receive any pans.
  3. You do not have any food.
  4. You burn dinner.
  5. You drop a plate while serving to end up with the picture above on your floor.

This is where error handling comes in.

The way we’ve been doing it is just to assign everything as an optional then handling it with an if let,if !(something.count > 0), or guard statements.

But what happens if you want a definitive answer now? There is something called a do/catch statement.

The format of the do/catch statement is do some bit of code, if something has the possibility to throw an error we will try to get the value that it returns or perform the action it does. If the method succeeds we exit out of the do/catch statement. If it fails, then we catch the error and give it a name so we can use it in the catch body using let error. What we call it doesn’t really matter, you can change error to whatever you want, just know it is of type Error. The Error property localizedDescription is just a computed property that returns a string description of what went wrong. You can omit 
let error if you want and Swift will automatically give you an error constant that you can use within the catch body.

I want to talk about try for a second. There are three different try options that you can use.

try — try to perform this function and return a result or prepare to catch an error if it fails

try? — try to perform this function, if it is successful return the non-optional value, otherwise don’t worry about any errors thrown.

try! — try to perform this function, I know it will not return an error and if it does, IT’S OK TO CRASH THE APP because I’m not handling the error.

So the first two options are great to have in your code, the third is really only useful when know the method won’t fail even though it’s marked as throwing. A good example of this is String(contentsOf:) when reading from a file included in your App’s bundle (the folder that contains all of your app’s functionality). At this point, it is the user’s fault for modifying the contents of your App’s bundle.

You might use this for checking file hashes to make sure the user didn’t change anything. If the user did, then we just make the program unusable and force them to re-download a new copy.

When you use try? or try! you do not need to wrap the code in a do/catch statement.

So how do we throw errors?

Errors can be thrown by appending throws after the parameters in your function declaration.

If you want to create your own errors, you can! You just need to make an enum that adopts from the Error protocol.

When we use throw we can use MyAppSpecificErrors.openFileError to denote that a file could not be opened. If we failed to open a file we might check to see if it even exists, even if we should have done so prior to opening it. If it doesn’t exist, we can create it and try again. If it does exist, then we need to report back to the user that they probably need to remove the lock (close the file) on the file we need.

With this way, we get some granularity on how we handle each specific error and what we do if we can’t handle that specific error.

Really you only need to write catch statements for the errors that can be thrown, but you must have a final catch statement that is for everything non-specific.

But what if we want to return the error string from a throwing function? There is an option for us and I hinted at it earlier. localizedDescription is provided for us for all of Apple’s errors, but it doesn’t just magically read the text fileOpenError and come up with the “Unable to open file” text. We have to tell it what to provide.

Here we have an enum that just covers FileErrors which adopts String and Error. We have to adopt String to provide string values for our errors.

Then we have two cases, each with their own assigned text, or rawValue. This allows us to create a computed property named something appropriate, in this case, I named it localizedDescription and gave it a type of String.

When someone calls FileErrors.kOpenFailure.localizedDescription they will receive the raw value of kOpenFailure which is "Unable to open file".

You don’t want to throw the localizedDescription, just the error, and when you call localizedDescription on the error later on, you will still receive the raw value. The raw value can be a String, Character, Integer, or FloatingType value.

Enums also have a nice feature that when you start with an integer for the raw value and it should scale, you only need to create the initial value and all others are automatically supplied with the next integer.

There will be more on Enums in the next lesson, but for now, we are just adding on to what we already know.

There’s one more thing you should know about throwing functions and that is the rethrows keyword.

Remember when we talked about closures and defined it as a function that takes another function as a parameter? If the function passed in as a parameter is a throwing function, we use the keyword rethrows in place of throws to denote this function will throw an error that was thrown by its parameter function. It would throw the error which would be caught in hopeThisWorks then hopeThisWorks would rethrow the error back to where it was called.

In the beginning, I told you that you had three options. Translated to your App let’s revisit those three options.

A. Let the app crash
B. Use a third-party library that may add more error-handling features than you need.
C. Write the code necessary to handle your errors.

Before this article, you might have gone with A or B. But I’m here to help you learn to fish. Now you know enough to go with C.

Summary

It is ok to write your program first without any error handling. Just mark each place where you might need to handle an error using the 
//TODO: Handle Error comment. This way you can go back through your code when you are finished to handle errors in a non-ideal environment. As you test your application, if you run into an error, try to handle it then.

So we learned about how to catch errors, how to throw errors, how to create errors and set descriptions for error cases, and how to re-throw errors. Not too bad.
