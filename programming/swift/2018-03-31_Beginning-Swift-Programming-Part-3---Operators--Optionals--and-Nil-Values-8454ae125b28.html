Preliminary Stuff

You should understand the concept of order of operations (PEMDAS)

In case math class was too long ago for you here’s the order:

  1. Parenthesis (from inside to outside)
  2. Exponents from left to right
  3. Multiplication from left to right
  4. Division from left to right
  5. Addition from left to right
  6. Subtraction from left to right

That’s it!

Optionals and Nil Values

Ok, I am going to try and make this easy but it might hurt your brain a tad. That’s good it means you will be learning.

I want you to think back on the first part of this series when I talked about variable instantiation and declaration. I kind of glossed over declaration and instead gave you examples of how we instantiated new variables. I’m not sorry, it was just to save you from this until you were ready.

That is perfectly fine, but what was the value of a before we gave it a value?

The value was nil. Not zero, but nil, zilch, nada, nothing! Oh Bob, how can a value be nothing?

For this, I want to talk about that image of the notebook that you saw above. Let’s say that the notebook has 200 pages in it, as we finish each page we tear that page out and continue finishing pages until we get to the last page. We tear out that last page. Now that notebook has 0 pages. All that’s left is the metal spiral that once held 200 pages and probably a cardboard front and back. Now we throw away that notebook. All that’s left is the desk that holds the notebook. That desk is our variable. The absence of a notebook means our variable is nil. We know that the desk holds notebooks that can contain any Integer value number of pages, but we didn’t give it a notebook so it is not instantiated.

Are you still with me? Let me repeat that again, the desk is meant to hold a notebook with pages, but we do not have a notebook on the desk. Let’s see how this look in code using a subtle syntax you haven’t seen yet:

A ? at the end of a type in Swift tells you the variable is an optional value. You cannot set non-optional variables and constants to nil. You can even instantiate optional variables up front.

What this does is it allows us to later set name to nil using name = nil.

You might be asking why would we want to initially set an optional variable to a non-nil value? If you populate a person’s last name in a form that came from a social media platform, you might create the variable lastName as an optional and provide a switch on whether or not to display it. If the switch is on, we continue providing the person’s last name to other users of our app. If the switch is off, we set the lastName variable back to nil. Later we display the value in lastName if it has a value.

Optionals are very powerful and later you will see how they make Swift a very safe language both in ensuring you are handling any cases where you didn’t get the data you expected and making sure you are using types in the correct way.

Optionals and nils also allow us to set default values using other operators. There are two types, ternary operators and nil-coalescing operators. We’ll discuss ternary operators in the next post, but for the time being, we will just go over the nil-coalescing operator starting with an example:

The nil-coalescing operator ?? is used to check if the value on the left-hand side is nil. If it is, we can set a default value on the right-hand side. In the example above age will be instantiated with the value 26 because we were able to get a value from possibleAge.

However in this example, age would be set to the value 0 because possibleAge was never set, making the left-hand side of the evaluation equal to nil. While this example makes it easy to understand, most of the time you will end up with many lines of code between where you set the value and where you ended up, so you’d really just see this.

let data: String = possibleData ?? "No data found"

Does possibleData have a value or not? What if we got this data from the web? We have no idea if we will have data or not until runtime, or when the application is running. The nil-coalescing operator really shines here because it allows us to handle the case where possibleData could or may not have a value.
