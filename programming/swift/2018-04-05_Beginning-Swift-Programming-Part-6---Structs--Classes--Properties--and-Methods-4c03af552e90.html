Structs

Structs (like structures) in C are data types that contain similar variables, in Swift, we extend this functionality to allow Structs to also contain functions, enums, and other structs, and they are declared like this:

We declare a struct using struct House, then in the body, we can put any number of variables, constants, or other structs as we see fit. If it makes sense for a variable to change declare it with var, if it doesn’t go ahead and make it a constant using let.

As I write out variables, I think about what it should measure or determine then I usually start with let and type the variable name. I ask myself a couple of questions about what type this variable should be. Here are just a few questions:

  • Does it make sense to use floating-point numbers (Doubles/Floats)?
  • Does it make sense to use whole numbers (Int)?
  • Does it determine a status (Boolean/Enum)?
  • Does it multiple things it can measure or determine (Struct)?

Asking yourself questions like these will help you figure out what types you want to use with a specific variable. squareFootage can be measured in whole numbers, but it can also contain decimal numbers, so we default it to a Double type. We can never have half a room or a quarter of a room so numberOfRooms becomes an Int, same for numberOfWindows. Why do you think I made numberOfBathrooms a Float. We would never have a quarter of a bathroom. We could have a whole bathroom, oh wait, we could have a half-bathroom consisting of just a sink and a toilet, or a sink and a bathtub. So we could return 1.5 for one and a half baths. Ok, that makes sense. Then we have address as a string constant. Why a constant? Well if you changed your address, 99.99999...% of the time it’s because you moved to a different house and this house would not apply to you anymore, so I think it’s safe to make it a constant. Then we have var doorIsOpen. Ok so this could switch between two possible value so we just make it a Boolean variable.

Of course, we can add a Garage to this struct, because we all know a garage can hold so many cars we can go ahead and say numberOfCars and make it an Int. Our garage may or may not have windows, so we allow for windows to be set but optionally using numberOfWindows: Int?. Finally we need to know if the door is open or shut if we are trying to pull out of or park in the garage. We do that with doorIsOpen.

Uh, Bob… we already have a variable named doorIsOpen and it’s a part of the base House struct. Yes, we do but because of scope the doorIsOpen variable inside of the Garage struct is not the same as the doorIsOpen variable inside of the House struct.

Let’s see how we can use this struct to create a house.

In case you didn’t catch it we use the dot operator . to get to the different elements of myHouse. If we need to read a variable from the Garage struct, we accessed the garage through the house. myHouse.Garage.doorIsOpen. Previously I said the door is open variable is not the same between the house and the garage structs, and the reason for that is myHouse.doorIsOpen is not the same as myHouse.Garage.doorIsOpen.

What’s that you say? You don’t have a garage, well that’s not a problem, instead of having nested structs we could break them apart and tell house it might have a garage.

Here we break Garage into it’s own struct, and where it was in theHouse struct, we replace it with a variable (maybe you might build one onto your house) named garage set to an optional Garage?.

Order matters if you want to do it altogether. We need to create a garage first if we want to initialize myHouse with a garage. Or because garage is optional, we could just set it to nil then later on create a newGarage and assign it to the myHouse variable like this myHouse.garage = newGarage.

Classes

Classes on the surface are exactly the same as structs. There is so little skin deep difference between them that people don’t understand why you’d choose one over the other. The reason everyone gives you is the absolute truth and it’s simple, Classes are reference types, Structs are value types. Pretty simple right? Aren’t you glad we went over reference types and value types?

Let’s be honest, that answer is simple, but in reality that answer is deeply complex. Classes are passed around by reference. Structs are passed by value type. Let’s see how this works between a class and a struct.

What happened to myHouse when it was used as a class? This is the implications of using a class. When we created the instance of myHouse we gave it an address in memory as a reference to House. The address referenced all of the information we needed for that instance of the House. When we created yourHouse, we gave it a different address in memory that contained all of the information for your house.

Here’s the tricky part. When we assigned yourHouse = myHouse we didn’t say yourHouse was like myHouse, we said yourHouse is myHouse, by pointing it to the address in memory that was referenced by myHouse. So when you shut the door on yourHouse, you really shut the door on myHouse. The information contained in memory for yourHouse was removed shortly after you assigned yourHouse to myHouse.

Now when we did the exact same thing using structs, we stored the value in the variable. When we assigned yourHouseStruct = myHouseStruct we might have well have changed an Integer variable from one number to another. Here’s the neat part about it. Structs are copy-on-write. This means that when we said yourHouseStruct = myHouseStruct, the system actually referenced the place in memory where myHouseStruct's info resided. You could follow it like yourHouseStruct = myHouseStruct = myHouseStructInfo. As soon as we shut the door on yourHouseStruct, a copy of myHouseStruct was made somewhere else in memory and yourHouseStruct was assigned that new piece of memory.

A class can also inherit from other classes. What this means is that any class that inherits another class, has it’s own variables and functions, but also adopts the variables and functions from the superclass, or the class it inherited from. Let’s see how this works.

Here we have a class named Ball it defines size, bounciness, and color. SoccerBall inherits from Ball using the syntax SoccerBall: Ball. This means that an instance of SoccerBall would contain not only secondaryColor but also size, bounciness, and color. Since Baseball isn’t the same as SoccerBall we wouldn’t want to have Baseball inherit from SoccerBall, but it is a type of Ball so we could again inherit from the superclass Ball to get the variables contained in Ball. Except now we have threadColor which comes directly from the BaseBall class.

Another way to look at this is just like in science where we classified organisms into Kingdoms, Phylums, Classes, Orders, Families, Genuses, and Species. Everything that is true in the upper classification of an organism is true all the way down to the species, however, things that are only available in a lower classification, are not available in the upper classifications.

Methods and Properties

I bet you’re happy to see that I made this all one section, I bet you’re also happy to know this section is short. But I’m going to tell you right now

I’ve been lying to you.

That’s right, I have, because as badly as I wanted to explain methods and properties to you, I didn’t want to introduce you to those names until you learned a little something about classes and structs.

Methods are just another name for functions when they are used in a class or a struct. The full name is instance method because they belong to an instance of a class or struct , but we just call them methods for short. I didn’t use any methods above, but I have previously in other posts. When I used the syntax var arrayCount = someArray.count, the array type has a built-in method called count whose only purpose is to count the elements in an Array so when we said someArray.count we just accessed the count method of the Array type. The count method returns an Int value so based on the rules of type inference arrayCount is an Integer type.

Properties are just another name for variables or constants belonging to an instance of a class or struct. There are properties of a House, a Ball, or even a SoccerBall.

Speaking of the fact that I haven’t used methods in a class instance let’s do that now talk about something new and then we will be done.

Here we access a method of myHouse. By using the dot operator and calling the function toggleDoor(), we can open or shut the door depending on the doors current state. We could check the status of door before we toggle it to make sure it’s open or shut by using 
if myHouse.isDoorOpen { myHouse.toggleDoor() } 
This shuts the door only if it is open.

And the very last piece of this section, the self keyword. self refers to the instance of this object. When we say self.isDoorOpen we are referring to myHouse.isDoorOpen but we use this because at the point where it is used, we have no idea what the variable name will be. It could have been yourHouse or theirHouse. Using self allows us to provide this function to all instances of a class or struct without having to worry about whether or not the syntax is correct.

Summary

We learned about Structs and Classes, how they are the same, and how they are different. There is a lot more to know about classes and structs but in Swift, use a Struct first and then move to classes only when you need the extra functionality that classes provide. When you do this you only need to change struct to class at the beginning. It’s that easy.

We also learned that I’m a terrible liar and that I wrote this entire post saying variable and function instead of property and function. (This was actually really hard to do and my delete key will vouch for me on this one). Just remember that a variable inside of a class or struct is called a property, and a function inside of a class or struct is called a method. Don’t worry if you slip up, you won’t get docked points for it, you have more important things to worry about like explaining to your boss what your method does and why you chose to use this type of property over another.
