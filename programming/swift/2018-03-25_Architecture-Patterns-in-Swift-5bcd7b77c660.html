Architecture Patterns in Swift

I know that you were probably expecting yet another post about Model View Controller (MVC), Model View Presenter (MVP), Model View View-Model (MVVM), or View Interactor Presenter Entity Router (VIPER), and I’m not going to talk about Clean Swift there are other articles already written for all of those that are very good. What I will talk about is what architecture is best for your app.

I hear a lot of talk about architecture patterns in programming, most of the time this all comes from people who have just started programming within the past year wanting to learn about which architecture they will use for their first app. My answer is MVC.

What! I came here expecting an immediate answer and I just got sent back to the MVC zone. What gives?

OK, OK, I’m sure I just gained a lot of haters from the seasoned programmers group but this is addressing the people just starting off, the seasoned programmers who just got all high and mighty, you should already know everything else I have to say and where I’m going with this but for the new programmers, stay tuned. I’ll be giving you a road map to help you move up and learn the various architectures.

I had read that it’s better to let your app decide what its pattern should be instead of just deciding up front what it should be and immediately start coding.

In case you have never run across them there are a few steps to writing an app from start to finish.

  1. Analysis — Write down your ideas, figure out exactly what your app will do, how you will make a living from it. This also involves discovering what needs to be changed in existing projects.
  2. Design — This involves wire-framing, creating image assets, mapping out how your app will function, writing out classes, functions, and variables that will be used and how they are connected. By the time you are done with the design, you should be able to give a flow chart to an outside person and they should be able to tell you high level what the app does.
  3. Development — Finally we can start developing, you may not realize it but as an App Developer, it feels like you spend more time doing everything but writing code. Anyways, this is where you *should* start writing out your code. Many people tend to start writing in Design, and others start in Analysis, but as long as you aren’t deleting anything you’re doing good.
  4. Unit Tests — In reality, you should perform unit tests before and after each small piece of code, but it had to go somewhere. Just imagine it as write a failing test, write the method that makes it pass, test again. You’ll be flipping between these roles a lot.
  5. Testing — like unit tests, you’ll be going between Development and Testing, if you are lucky, a lot less. Testing is where you run the app just to make sure everything is working as expected, your views are generating correctly, your variables are being set properly, and your data is in the format you expected it to be.
  6. QA/UAT — Quality Assurance is where you send your project off to people who only have one purpose in life, to crash your app. This is a good thing, it helps to find the edge cases where you failed to catch errors or normalize data. User Acceptance Testing, on the other hand, is where you send this off to people who just use your app and expect it to work. The object of this game is QA fails a little if at all, UAT doesn’t fail.
  7. Release — If you made it here, congrats. This is the point where you start making money from your long nights and countless weekend hours of hard work. Time to start all over again for the next update.

In the analysis stage start small, think MVC. If you realize you are going to do the UI entirely in code switch it up to MVP. If the project gets to a point where it’s getting hard to read, refactor your workflows to MVVM. You should only need to go to VIPER if you are writing some serious enterprise-grade stuff (think Quickbooks, SolarWinds, Rapid7, Nessus). Ideally, you’d have this figured out before you start development, but as a new programmer, you’ll probably refactor during development.

It doesn’t mean you can’t use VIPER for a something as simple as a networked Todo app, it just means you’ve now spent more time writing the VIPER boilerplate code than you did your own code for the logic for the App’s primary functionality. If you get to the point where you need MVVM or VIPER you’ll be happy you did because the maintenance is a breeze.

Continuing on the idea of a simple networked Todo app, let’s say that it does a lot more than just save your Todo lists, what if it expands on Todo lists and adds in a calendar feature, maybe allows invites, works with Facebook, includes synching with various providers, maybe provides a Gantt chart for those super organized folks. You might consider using MVVM or VIPER up front, even though it really only requires MVC or MVP for what you are writing. Adding in those new features would be comparative to adding a new Cocoapod to your project. Even if you decide I’m going to swap out AFNetworking for Alamofire or vice versa, you only have to change the parts of the functions in the model that handles networking.

In summary, I’m only bringing this up because I see a lot of new people who say “I want to learn Swift” two days later they say “I want to learn VIPER.” To these people, I’d say, “Learn Swift using MVC until you are so good with the API that it almost comes naturally, then learn delegation and dependency injection (DI) followed by Protocol-Oriented Programming (POP)” Between delegation, DI, and POP you should have your app mostly decoupled. Then you can start focusing on learning MVVM or VIPER.

Swift is declared as a Protocol-Oriented Programming language. Even Apple says “First, start with a protocol” and I do not disagree. For the beginners out there, stop worrying about which architecture you should use, know the language and most of it’s commonly used classes and structs first.
