Protocols have a lot more power, even more so with extensions, but we will cover those in the future. Given that you now know the commonly used top-level entities in Swift, I figured it would be the right time to talk about code structure and readability.

If you have been practicing like I hope you have, you’ve probably made a sizable program, had dinner or went out, and when you came back you felt a little lost in your own code.

This happens a lot when starting out. You get frustrated if you are distracted because you forget what you wrote down 5 minutes ago. Don’t worry, it’s happened to most of us. I have good news, it does get easier over time, how long that takes depends on how much you practice.

For me, it took about 6 months of everyday coding. From the time I woke up to the time I went to bed, even on weekends, I was always writing code. Don’t get me wrong, it didn’t mean that I didn’t have a life, I still went out and had fun. Sometimes I’d burn myself out and have to stop writing code for a day or two. Even then I was still reading articles and trying to get better at what I did.

So keep practicing.

Code Structure

The other day I started thinking, there really isn’t anything out there about code structure in Swift. Everything seems to be hearsay or following the structure of a tutorial. Everyone has their own quirks with their writing style, there is nothing standardized. I am not trying to standardize code structure. I want to make that clear. I’m just taking a conglomeration of everything I’ve seen and my own code styles to help provide a starting point for how you can keep your code organized and easily understand where things are when you come back to your code later on.

With that being said, let’s talk high level about how various parts of an application show up in your Swift files.

Classes, Protocols, and Extensions are exclusively top-level entities. By top level, I’m saying they are not surrounded by curly braces. They sit against the bare file so to speak.

Structs and Enums are almost always top-level entities, however, they can also appear within other structs or enums.

Functions can be top-level objects but I would only suggest doing this in Console programs. For macOS, iOS, tvOS, and watchOS programs, I’d stick to keeping functions within classes.

Constants and Variables should be declared at the lowest scope possible, this means if you only need a variable to exist in an if statement, declare it inside the if statement. If either will be used throughout a file, declare it at the top of the file and provide the appropriate access control.

Readability

Readability is more important for your own sanity as well as the other developers that read your code than it is for the user. You should strive to have great readability in all of your code.

Breaking things up

As you can see these files get pretty long when you have so much functionality written into it. So what I tend to do is break these up into separate files under the same folder group. The naming convention that I’ve seen used is as follows: 
ViewController+NameOfDelegate.swift or ViewController+NameOfDataSource.swift

This way I know what code should be included in the file I’m about to click on and it helps me to make sure that file only contains the code that pertains to it.

Using the example code above, I’d rip out the enums and place them in their own file, if I wanted to create a single file named Enums.swift for global enums, I could, then I’d use //MARK: to split it into sections so I could jump back in for updates later. If the enums in the file were only for this view controller, I’d keep the access control set to internal but I’d create the file under the same group as the ViewController class.

I might take out the protocols and their extensions and place them in their own file unless the protocol was defined in a model and was meant to be used by other classes. If the protocol had class or struct specific extensions I would use //MARK: to separate them logically through the Protocols.swift file. Again I’d place the files in the same way that I would Enums.swift.

I’d also rip out the struct and use it as a model named House in, you guessed it, House.swift. I’d also separate it out into a group named Models, because it is a model for a house.

The class and all of its methods would stay the same, but I could take out the class-specific extensions and place them in a new file ViewController+Extensions.swift. Just make sure that any methods in ViewController.swift that need to be called by an extension are set to internal, sorry but private and fileprivate will prevent you from seeing those methods from an extension when the extensions are placed in another file.

Constants are special. One trick I see people do is create a Constants.swift file and place all of their constants that are used throughout the app in there. That way if a URL that your app needs changes, you only need to change it in one place and you know where that place is immediately. Then if you use it in 5 different places, they can still continue to reference the constants file and you can be assured they are up to date. One change vs five. Sounds like a done deal to me too.

Here is a great post providing more information on using a constants file on Stack Overflow. Be sure to read further down where a user explained how you can use enums as constants.

Using descriptive variable names

Don’t try to obfuscate the code by using one letter variables like people tend to do in C/C++. Over the years, i, j, k, l, m, n, t, x, y, and z have had so many values for so many things, they could mean anything. By telling me you have a program that calculates distance traveled at a given speed over a period of time using nothing but j, k, and m, says absolutely nothing, that’s all great senior dev, but what is k. Senior dev looks at the code for a minute… Ah, k is the speed variable.

Really? Who would have ever known that k is the speed variable without having to think about what the formula the senior dev used to write the program? Isn’t that the point that we should be able to instantly look at a section of code and say yes this is where the speed is set, I can add a 5 minute break here to simulate a stop and be done in the same amount of time it took for the senior dev to tell me what k was.

A little history on single letter variables, the reason why developers used single letter variables was that autocomplete didn’t exist and developers didn’t want to type as much as they had to. It also made sure that a programmer didn’t misspell a variable name. It’s about the equivalent of texting on a phone in the early 2000s, we used lol, c u l8r, < 3 u, and ttyl. To shorten our words so we only had to push a button 20 times instead of the 100 times if we actually spelled out the full word. Times have changed, almost every major programming language has autocomplete.

Sometimes it’s ok, for example, temporary variables in if statements where you can still see the declaration 5 lines away from where you are typing. This is like using lol on an iPhone. All is good. But when you use a as a variable used throughout a class or struct, well then it’s a problem. I shake my head like I do when I see someone type c u l8r on a smartphone. But didn’t you have to change the keyboard type to hit the 8 key? Seems like more work than it’s worth. Same way with the global single letter variable, you have to figure out what it stands for before you can use it in your changes 8 months later, seems like more work than it’s worth.

If you forget everything I said there, just remember to always use descriptive variables and constants. I’ve been writing out these tutorials from scratch without autocomplete typing in some pretty long variable names with very little use of copy-paste. It shouldn’t be a problem for you to give your variables descriptive names with autocomplete.

White Space

White space is always good for code readability. When you try to masheverythingtogetherandtrytocomprehenditlateronit’sreallyhard.

Instead, use white space. It keeps your frustrations down when going through your code and it helps separate your code into logical subparts.

As you saw in my example code above, I used white space where it made sense.

  • I had bold use of white space in between each mark by using two blank lines.
  • I put a separation between each function.
  • I used it between variables that played a different role in the class.
  • I put a space between the class and function declarations and the opening brace of each respective body.
  • I used it right before the end of the class as a subtle hint the class body was over.

All of this helps make my code more readable. David Heinemeier Hansson, the creator of Ruby on Rails, gave a speech at a RailsConf, explaining that very few programmers are computer scientists. Instead, we are better named as software writers, because that’s what we do. We write software. You may not agree with me and you may still want to cling to the title of computer scientist. That’s fine, but I strongly suggest that you watch the video of his speech. Then decide for yourself. I provided a link instead of an embed due to language content. The video later talks about Test Driven Development (TDD). It’s a good bookmark for future reference, but we’ll be talking about testing near the end of this entire series.

Next thing, place all functions in your class first by their visibility, then by the order in which they are called. Unfortunately, if your code is not encapsulated in a class, you will need to place all of your functions before the code where each function is referenced. The best guide for this is to place your functions at the top, with the functions that do not depend on other functions at the top followed by the functions that do depend on those functions. In essence, if a calls b and b is dependent on c and d. the order from top to bottom would be c, d, b, a.

The last bit about readability that I want to talk to you about is this: Keep your related code near each other. It doesn’t mean force all related code to run one after another, it just means that if you are using data from the web, all of your networking methods should be grouped together, all of your web focused variables should be placed together, and any logic to retrieve the data using the variables should be placed together. Even if the variables are at the top, the logic is in the middle, and the web service code is at the very bottom.

Principles

Previously we talked about DRY (don’t repeat yourself), Separation of Concerns, and the Single Responsibility Principle.

I’d like to expand on that and talk about some others.

First, “if it ain’t broke don’t fix it”. This should be a pretty straightforward principle. If the code you wrote works, even if you have no idea how it does, leave it alone. Chances are if you try to fix it, you’ll just break it and not know how to put it back the way it was.

Information Hiding. This is simply making sure that the accessibility (defined by access controls) is as restrictive as possible on every piece of code you write. If no one else is supposed to see mySuperSecretFunction() don’t make it public. If you have networking code that builds a URL for you, let the networking class build the URL, don’t have your view controller do it, set that method to private or fileprivate in the networking class.

Loose Coupling. This sounds abstract but it’s really not. When you have a class use part of another class, just get only what you need from the class, don’t instantiate that class inside of your original class otherwise you get a strong reference. Loose Coupling is just the result of keeping your dependencies between classes low. Use structs, delegation, or protocols to help lessen the coupling between objects. Delegation hasn’t been covered but in short, it’s allowing a class to become a delegate, or act on behalf of, another class by allowing certain functions to be exposed to the delegate class.

Law of Demeter. I pulled this info directly from Wikipedia. You should want this to be true for everything you write in code.

  • Each unit should have only limited knowledge about other units: only units “closely” related to the current unit.
  • Each unit should only talk to its friends; don’t talk to strangers.
  • Only talk to your immediate friends.

And Finally, for now, the open/closed principle. All entities should be open for extension but closed for modification.

  • Classes can be closed off from inheritance using the final keyword before the class keyword.
  • Variables can be turned into constants where possible then the value can be used in variable later on if it needs to be changed. The constant will still be the same value.
  • Use extensions to create custom logic for specific functionality instead of updating the original entity with the custom logic.

Summary

We took a look at how your code can be laid out so you can come back to it later and not be totally lost. We also looked at how you can take that code structure, or your own, and make it more readable by using new files, white space, and descriptive naming conventions.

Code readability extends even further including how methods are named. But I figured I’d save that for a naming convention section sometime in the future since there are so many different cases.
