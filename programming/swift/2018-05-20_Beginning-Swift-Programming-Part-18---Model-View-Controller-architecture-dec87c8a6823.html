You’ve done it, you’ve learned the basics of Swift. Congratulations. Now you can move on to actually writing your own apps. It’s at this point where you ask yourself a couple of questions.

  1. How do I implement what I’ve learned to make an app?
  2. What should I learn next?
  3. Company A is looking for an intermediate Swift developer, and I’ve only just learned Swift, is it for me?

We will cover the first two questions in this article. This article is written to teach you how to use Model View Controller (MVC) architecture efficiently. In the end, I’ll provide a list of my favorite resources for learning Swift.

As for the third question, this depends on your experience in programming. If you are coming from another language, you should already know the answer to this question but I’d say you’d be at the entry level until you’ve written at least 20 programs (or at least 1 full-featured app) in Swift.

If you are new to programming and this is your first guide, you should not apply. Even if you get the job, they are going to ask you to do something you haven’t learned and it is going to be very hard. Something like, “I need you to download data from this URL, parse the JSON response, sort it, then save it to a plist file in the provided format.” You’re just going to give a blank stare. Don’t do it.

One more thing… I know there are websites out there such as HackerRank and CodeWars that help you with your programming skills, but these primarily focus around algorithms. There is nothing wrong with these websites but think about the last time you needed to use a Euclidean algorithm to do your taxes, read a text file, or parse the results of a web response.

While there are some things that are good to know such as linked lists, queues, stacks, binary trees, tries (pronounced trees). I’m not bashing those websites, they are fun and good ways to improve when you aren’t working on a project or if you want to experiment with another language.

Model View Controller

MVC is where almost everyone starts when writing their first app. It’s easy to use, easy to understand, and keeps your code nice and tidy while providing clues on how to decouple your code and keep it clean.

So let’s talk about architecture for a second. When we hear the word architecture most of us tend to think of buildings and bridges, this isn’t to abstract from what we are talking about with programming.

The architecture of an app doesn’t just define the folder structure of your project but also defines how each class interfaces with other classes and modules. For instance, it would seem necessary to allow a file retrieval class to work with classes that manipulate the data within the files. But it may not seem good for a file manipulation class to work with classes that are responsible for laying out the user interface (UI). There are too many degrees of separation.

The idea behind your architecture should be that entities should only be visible to the fewest number of elements as possible. Views should only be responsible for UI work, Models should only be responsible for storing, formatting, and retrieving data, and Controllers should only be responsible for routing data from the Model to the View.

Views

Views are used to hold logic that displays data to the screen. Whether this is a custom drawn shape, a custom table view cell, or a custom button appearance. This is the code that holds the drawing logic and the outlets (IBOutlet) that the view will use. Typically views subclass from UIView, UITableViewCell, orUIButton, but they can be anything that is only meant to display objects to the screen.

Controllers

iOS has a bad rap about having massive controllers and they do get pretty large. Controllers are where the logic occurs in your app that takes data from your models and provides them to the views to be populated. The idea is that when done correctly the view has no idea the model even exists and the model doesn’t know how it’s data will be presented.

Controllers in iOS are indicated by the ViewController suffix appended to the name. SearchViewController is a controller, SearchView is a view. Items might be the model.

MVC

Referring back to the phone book example, we have a phone book as our model. It contains an array of entries. Our controller might be responsible for declaring and instantiating the Phone Book and pulling whatever data out of the phone book entry that the view will need. After the data is retrieved, we then populate the outlets inside of the view and present the view.

If the user changes the phone number for a person, the view notifies the controller and the controller handles sending the data back to the model by calling one of the models update methods passing in the changed data.

The update method inside of the Model class is responsible for formatting the data so it can be stored for retrieval later on. This means if the user’s phone number was in the format 123.456.7890, we need to strip out the . so we can retrieve it later in the format we specified to keep everything uniform.

The view’s only purpose is to display information to the user, in iOS views are pretty dumb. They don’t do a whole lot outside of displaying the UI and notifying the Controller.

The model’s purpose is much like a secretary, it just keeps the data in order. It has a very important task of maintaining the data. Models do not permanently store data, but they can be a reference to provide the format of the data that will be stored in a database.

The controller’s purpose is to pass data between the model and the view. It seems easy enough, but sometimes there is a lot of logic that needs to happen in order to do so. For instance, the controller will need to hold a reference to every model that the view it controls will use and use each classes methods to perform updates and retrievals.

Summary

Don’t let your models talk to your views. Use views to display content to the user, use models to store data, and use controllers to handle any intermediary logic between the two.

MVC is a good starting point for every app you write. Yes, there are other architecture patterns that you can use but this is the easiest to use while starting off. You might be tempted to try something different but I promise you, they are all more difficult.

As I put it in my first article on Medium, let your app decide what architecture you should use. Don’t say “I’m going to write an app using _______ architecture.” When people say something like that I cringe because a majority of the time the app they are trying to write doesn’t require anything more than MVC. This doesn’t mean don’t ever leave the option out. I’m just saying start with MVC as you are figuring out how your app is going to work and switch to a different architecture if the benefits outweigh the amount of work.
