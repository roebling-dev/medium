inout

inout is a keyword that is used when passing parameters into functions. inout is used when we want to pass in a variable to a function and modify the value of that variable without creating a new variable.

Let’s take a look at how we would modify a value with and without inout.

When you create a standard function without using inout the variable passed in is immutable, or it cannot be changed. In other words, it is passed in as a constant. inout allows us to change the variable passed in because it is passed in by reference instead of by value denoted by the preceding &. In case you missed it, I wrote about this in part 2.

In the example above when we used a function without inout, as number was passed in to multiply(number:by:) it was passed in not as number but as the current value of number, or 10. We might as well have called it like this: 
number = multiply(number: 10, by: 2).

Looking at the function that uses inout, there are three changes, the first is we do not have a return type. The second is we use the keyword next to our parameter type inout Int. The third is that we do not have a return statement in our function body.

When we call the inout function we do not have to assign a return value because it doesn’t return a value. Instead, as we call the inout function we are actually passing in the variable’s location in memory. For the sake of clarity, let’s say that secondNumber was assigned to memory location 0x01 when we used &secondNumber, this was translated to look like the following: 
multiply(number: 0x01, by: 2). Don’t be confused, because when we look at 0x01 to see what the value is, it’s still the value of secondNumber: 5.

Inside of the inout function things work a little different, we use 
number *= multiplier to change the value stored at the address of secondNumber. Because we change the value directly the changed number is reflected everywhere secondNumber is referenced in your program. The upside to this is you have a smaller memory footprint and you only have to worry about just this variable containing the value of number.

There is a downside to using inout and it plagues all reference types. If secondNumber is changed while it’s being referenced elsewhere in your program, you may not want the value to change everywhere.

As a more practical example, what if we passed a value in called userData that was of type Data and it contained some important information that we needed to reference somewhere else in our app. We passed it into a function as an inout parameter and the data changed while we still needed the original data. We would refer back to userData expecting it to be one thing but it had changed. Best case is that our app can handle the change and continues chugging along. Worst case is that our app crashes because the value was nil or maybe the data type stored at that location in memory had changed.

It’s up to you on whether or not you want to use inout. Even though it’s not completely safe, it doesn’t mean you shouldn’t ever use it. It works great when you are performing calculations and you don’t want constantly assign the result of some function to currentResult every time you perform a function in your calculation.

Lazy

When we create a class, we almost always create properties used by that class. These properties may just be a flag that we turn on or off to determine the current state of the class or they might be something bigger like another class that our new class uses to do perform some action.

Let’s look at an example, I promise it’s related.

This example above doesn’t seem like it’s a huge issue, using this class just uses a CLLocationCoordinate2d to show a location on a Map.

In case you were wondering, a CLLocationCoordinate2d is a struct that contains the latitude and longitude of a location along with some basic methods. Both the latitude and longitude are CLLocationDegrees which is just a typealias for theDouble type. Keeping it simple, CLLocationCoordinate2d is a way to provide two Double values to find a location on a map. It’s pretty lightweight overall.

The MKMapView on the other hand, takes up a lot of memory. Just loading up a map and zooming into a coordinate takes up a couple of hundred megabytes. When I was using a map in one of my own programs, I loaded up almost 2,000 annotations and after panning around a little the memory usage jumped to about 430 MB. So as you can see, map views get a bit expensive. And yes, the map view is the same map you see when you open the Maps app.

The good news is iPhones and iPads come with a couple of gigabytes of memory so this isn’t a huge deal, but it can still be optimized. This is where lazy stored properties come in.

I’ll give you a scenario, you have a view, this view could show a map view, or maybe not. If it doesn’t, we definitely don’t want to load the hundreds of megabytes into memory but we still want to be able to if it will be displayed and make it available to be modified in other functions owned by the view.

Let’s take a look at how this would work.

The lazy keyword in front of var mapView = MKMapView() is where the magic happens. This tells the program, be prepared that a mapView may be used in this view. When the time comes to use the map view, here’s the code to create it.

Further down when the user taps a button to display the map view, we call createView() which contains the logic we wrote behind the scenes to add the view that will show the map to the current view. When we use view.addSubview(mapView) the mapView code is called which creates the mapView and we can call functions on the mapView as needed.

If the view wasn’t created and we called a function on the mapView, the mapView would be created at that point. So lazy basically delays creating the mapView until it’s absolutely necessary.

Lazy properties can also be closures which is how they are most often used to hold off on extra computations until they are absolutely necessary.

You see this a lot in a Core Data stack. Don’t worry if you don’t recognize anything, we are only focused on lazy.

Persistent containers may or may not be needed, but we don’t need to create it directly up front. Instead, we wait until we actually need to retrieve or store data in our database before we create it. This uses a closure which creates the persistent container with the minimum steps needed to create the container.

Lazy is nice to use and it should be used often where it makes sense. Don’t worry about making simple counters lazy. And, by the way, you cannot make constants lazy. If you try to make a constant lazy, Xcode will get upset with you.

Getters and Setters

Getters and setters are a part of computed properties. They are close relatives to the property observers didSet and willSet. As you may remember didSet and willSet are there to perform extra tasks when a computed property is about to change or has changed. Getters and setters provide us with logic that we can use to set the value or retrieve the value. Here’s an example

In this example, number is provided the default implementation of get and set that we would have if we didn’t add { get set } to the end. The only reason we do it is for clarity purposes. It’s always good to be consistent and if you have a property that is get only, then it is in your best interest to add 
{ get set } on properties that are gettable and settable.

When we only use { get } we are only allowing the value to be retrieved. We cannot set the value. It’s similar to a constant however it typically returns a computation of other properties.

Let’s look at a few examples where we would use { get }.

In this example we have an hourly rate and number of hours worked which can be set using 
myEmployee.hourlyRate = 9.75 
myEmployee.hoursWorked = 40

Then we can retrieve the employee’s earnings by using 
let wagesEarned = myEmployee.earnings

Simple enough, let’s kick it up a notch.

I left a few extra spaces to make it easier to read, but in this example, we have a private setter denoted by the beginning private(set). This allows us to make sure the proper value is set. We’d never have a negative earnings or 0 hours, so we can make sure earnings was set to a positive value and we at least worked sometime before we actually set earnings and hoursWorked. Then we use the setter of earnings to update the value of hourlyRate.

This code works for updating the earnings of our employee and the number of hours worked, but what if the employee gets a raise? You could create another updateEarnings(to amount: Double, rate: Double). But this would also mean that you’d need to remember if you set the hoursWorked or hourlyRate so you could handle which one needed to be updated.

If you end up in a position like this getters and setters might be a bad idea and instead, you might switch to the property observer didSet for both hoursWorked and hourlyRate. You could still keep earnings to a private setter, but it would be easier on your sanity to not try and update the values using the private setter.

Using the code above, see if you can refactor it, that changes it, so it can use two functions to update the earnings and hours or rate to make this Employee struct a little more usable. You could even use it to compute changes to your own pay. Maybe add an isPaidHourly flag to automatically set hours to 40 if it is false (for salaried workers) and have a method to update the earnings for salaried workers.

Tip: while you could use optionals and use a method declaration like this: updateEarnings(to amount: Double, hours: Double?, rate: Double?) 
It is better if you break this up to provide a more intentional update to the amount and either hours or rate and let the property observers perform the updates for you.

The other way I’ve seen private setters be used is through computed properties. Such as the square of a number that is performed when the number to be squared is set.
