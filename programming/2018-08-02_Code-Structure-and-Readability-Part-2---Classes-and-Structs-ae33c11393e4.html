Classes and Structs

Let us start by throwing an example class out there.

public class Animal {
    let name: String
    let favoriteFood: String
    var age: Int
    let countryOfOrigin: String
    init(name: String, favoriteFood: String, age: Int, 
         countryOfOrigin: String) {
        self.name = name
        self.favoriteFood = favoriteFood
        self.age = age
        self.countryOfOrigin = countryOfOrigin
    }
}

We could, of course, change this out for struct and forget the initializer but I’m going with a class because let’s face it, it’s a bit sloppy even when you try to make it readable.

To be completely honest, I try not to use a class when a struct will do. I don’t have to worry about initialization (unless I want to), I don’t have to worry about reference cycles (unless I’m going for extreme optimization) and my code base is a few lines smaller.

So what can we do with classes to help out with this problem? Well, spacing sure does help so let's start there:

public class Animal {
    let name: String
    let favoriteFood: String
    var age = age
    let countryOfOrigin: String
    init(name: String, favoriteFood: String, age: Int,
         countryOfOrigin: String) {
        self.name = name
        self.favoriteFood = favoriteFood
        self.age = age
        self.countryOfOrigin = countryOfOrigin
    }
}

A little bit of spacing went a long way with this one. But I think we can do better than this. Whenever my code appears hard to read like this part:

...
    init(name: String, favoriteFood: String, age: Int, 
         countryOfOrigin: String) {
        self.name = name
        self.favoriteFood = favoriteFood
...

I try to add in a bit of extra space so I can see that I’m assigning name to the class instance variable name. Like this:

...
    init(name: String, favoriteFood: String, age: Int, 
         countryOfOrigin: String) {
        self.name = name
        self.favoriteFood = favoriteFood
...

This makes it a whole lot easier to see where my init declaration ends and my assignments begin. Let’s see what this looks like all together.

public class Animal {
    let name: String
    let favoriteFood: String
    var age = age
    let countryOfOrigin: String
    init(name: String, favoriteFood: String, age: Int,
         countryOfOrigin: String) {
        self.name = name
        self.favoriteFood = favoriteFood
        self.age = age
        self.countryOfOrigin = countryOfOrigin
    }
}

Looks a little better but now there is another issue. The init(name... line looks out of place, I’ll add another space to differentiate between the properties and initialization. While I’m at it I also don’t like how we have a variable var mixed in with constants let, I’ll fix that too.

public class Animal {
    let name: String
    let favoriteFood: String
    let countryOfOrigin: String
    var age = age

    init(name: String, favoriteFood: String,
         countryOfOrigin: String, age: Int) {
        self.name = name
        self.favoriteFood = favoriteFood
        self.countryOfOrigin = countryOfOrigin
        self.age = age
    }
}

Notice that I keep my initialization parameters and my initialization code in the same order as my properties. This way, when I go to make a change in my class or just review the code for that class, everything is easy to find.

But the more I look at this the more it just bugs me. We need to refactor a little. name and age are short variable names but favoriteFood and countryOfOrigin are seemingly long. While they are descriptive names I think it’s safe to shorten these names a little without losing the context of each belonging to anAnimal.

public class Animal {
    let name: String
    let favFood: String
    let origin: String
    var age = age

    init(name: String, favFood: String, origin: String, age: Int) {
        self.name = name
        self.favFood = favFood
        self.origin = origin
        self.age = age
    }
}

Because of that refactoring, not only did we make the properties jive well together, but we also shortened our init function enough so all of the properties fit well together on a single line.

Comments are still important even with self-contained classes. Typically I like to break my classes out into sections.

// Class definition
// Properties
// Initialization
(// Static methods)
// Public (methods)
// Private (methods)

Everything in parenthesis is optional text or where it makes sense. Another thing I tend to do is put the most visible properties and methods before the least visible properties and methods. I always insert my init function directly after my last property and before all of my methods. I always place static functions directly after init because most often it will be used internally or be a public API. If I don’t have a section I just omit the entire comment. So let’s look at this again. I’m also going to change age to a private property with a public getter method because I can’t just say Animal just got 10 years older, instead, I need to let Animal have their birthday.

// Animal
// Defines an animal with a name, favorite food, and country of 
// origin.
//
// name: Gets or sets the name of the animal
// favFood: Gets or sets the animals favorite food
// origin: Gets or sets the animals country of origin
// age: Gets the animals age
public class Animal {
    //Properties
    let name: String
    let favFood: String
    let origin: String
    private var age: Int

    // Initialization
    init(name: String, favFood: String, origin: String, age: Int) {
        self.name = name
        self.favFood = favFood
        self.origin = origin
        self.age = age
        sayHello()
    }
    
    // Public
    func hadBirthday() {
        self.age += 1
    }
    func getAge() -> Int {
        return self.age
    }
    
    // Private
    private func sayHello() {
        print("Hello, World!")
    }
}

At the top, you will see what is known as a docstring. Sometimes this is above the class declaration like in the Swift example above. Other times it will be below the class declaration such as Python.

class Animal:
    """
    This is a docstring in Python. Best not to go over 80 characters
    or PEP8 will scream at you.
    """
    ...

80 characters are all PEP8 wants you to use for your docstring but it’s a good number for any language in my opinion.

Don’t just make documentation because I said to, do it for the poor soul who has to make changes to that code after you. It will happen, business rules change, having the documentation for them helps them understand what it does and makes their life a whole lot easier. This may not help with readability, but it helps with navigating through brand new code. Sorry took a detour, but it was important! Let’s go back and look at the code:

// Animal
// Defines an animal with a name, favorite food, and country of 
// origin.
//
// name: Gets or sets the name of the animal
// favFood: Gets or sets the animals favorite food
// origin: Gets or sets the animals country of origin
// age: Gets the animals age
public class Animal {
    //Properties
    let name: String
    let favFood: String
    let origin: String
    private var age: Int

    // Initialization
    init(name: String, favFood: String, origin: String, age: Int) {
        self.name = name
        self.favFood = favFood
        self.origin = origin
        self.age = age
        sayHello()
    }
    
    // Public
    func hadBirthday() {
        self.age += 1
    }
    func getAge() -> Int {
        return self.age
    }
    
    // Private
    private func sayHello() {
        print("Hello, World!")
    }
}

Between the public properties and private property, I separated these. If I had more private properties, all of the private properties would be grouped together, as long as they made sense to group together. If there were related properties, like birthday and age, I would group these together separate of the rest of the private properties. Since all of my public properties in this example are unrelated, it doesn’t hurt to put them together. File this under miscellaneous properties if you will.

Moving past the initialization for a second (not much has changed anyway), we have public and private method areas. Public sits before private because there is a good chance you will change the public methods before the private methods simply because they will be called more often than your private methods (in most cases) and bugs will be more apparent. Notice here we can actually get at the private property, this is what other developers will call to use that hidden property so to speak. We don’t want them directly accessing age. No one would like it if you suddenly said they were 75 and the whole world agreed with you unless in fact, they were 75 or older.

Now we have a private property, this just says “Hello, World!” and that’s it.

Going back to our initialization, we added this call, but if you notice, I added the call at the bottom. Sometimes it makes more sense to place it nearer the top, like when it does some calculation to instantiate a private variable automatically, but here it’s fine to place it at the bottom. We just call it so when the class is fully initialized, it lets us know by welcoming itself with “Hello, World!”

If you look at this closely, we didn’t lose any readability, the code is clean, and its structure is pretty sound. Yeah that docstring is loud but not much we can do about that until we have color:

See how much better this looks? Well defined whitespace, noticeable sections, this code looks pretty clean.

Summary

So this was a pretty short article but I hope it got the point across on how you can structure your classes and structs to make them look much better than before. Classes and structs get much larger than this, and you can usually find some nice clean code out there on GitHub.

Microsoft typically has some good coding practices but I’ve also seen really good code from Apple, Google, and others.
