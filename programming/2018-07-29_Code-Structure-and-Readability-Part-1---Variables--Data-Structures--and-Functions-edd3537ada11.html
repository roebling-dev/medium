Code Structure and Readability Part 1 — Variables, Data Structures, and Functions

If you came here from my Swift programming article on structure and readability, you already know some of the basics. If not, that’s ok. I’m going to go over a few of the topics that were mentioned and expand on those here. I recommend that you at least scan through the article above for some quick tips.

First and foremost, Swift is my primary language, followed by Python, COBOL, C#/Java, then C/C++. I’m sure that even if you use a different language, these tips will still apply.

The format of this series will be as follows:
1. Variables
2. Data Structures
3. Functions
4. Classes/Structs
5. File structure

We will be starting at the most basic ways to keep your code clean and expanding from there.

Overview

At one point in your programming career, you may have come across a program that looks similar to this:

#include <stdio.h>
#include <stdlib.h>
int main(int argc, char *argv[]) {
    int n1,n2,s;
    n1=atoi(argv[1]); n2=atoi(argv[2]); s=n1+n2;
    printf("%d\n",s);
    return 0;
}

How long did you have to look at that program to figure out what it does? Probably took a second right?

In case you never took a course in C, atoi is a function that converts an ASCII string to an integer. The rest should be pretty familiar.

Let’s look at this example again.

// add.c - takes two integers passed as arguments and prints the sum
#include <stdio.h>
#include <stdlib.h>

int main(int argc, char *argv[])
{
    int first_number = atoi(argv[1]);
    int second_number = atoi(argv[2]);
    int sum = first_number + second_number;
    printf("%d\n", sum);
}

See how much easier this is to understand? You can just glance at it and get an idea of what it does. So what changed? This is what we will be covering throughout these topics.

Variables

Many times I’ve been browsing through code and came across something like s=n%k. I’m not so worried about the lack of whitespace, but what does concern me is what s should be. In order to know that we need to know what n and k are. This is a terrible design.

To make matters worse, when the original developer goes to modify this program later on with another variable that is similar to s, they use s1. Do you see where this might be a problem when you have many variables with very similar names that might mean very different things?

When writing variable names in Swift, I tend to use descriptive names. The rule I use is the shortest name that describes exactly what this is. It might be as simple as sum in a local scope, but in a more broad scope I might use total or sumOfAllCars. For python, I’d use snake_case instead of camelCase as it would integrate well into external libraries.

Naming conventions should work well with the language, not just your program. Always have a good understanding of how other developers write out their variables in your language and mirror it, but be descriptive. No one likes to look at sum_ra_EE as a variable name. It doesn’t make sense. Who would have guessed that EE was the module this belonged to? And what exactly is ra? I’d have to browse through almost half of your entire code base to figure out exactly what ra stood for. If this were a public interface to my library, it would turn away a lot of developers simply because it isn’t intuitive.

Clean readable useful code = happy developers = more github stars = more support = more use = more donations. And who doesn’t like money?

So when naming variables, make them understandable to someone just looking at your code for the first time. Provide some idea of the intent for this variable within the name.

Going back to the first example above int n1,n2,s;. At first glance, this looks to be pretty understandable and in simple programs it is, to be honest, it was just fine the way it was. But as your program gets larger we end up with n3...n47. When you want to use a variable in the future, you don’t want to be scrolling around while asking “What was n26?”. This is why proper naming is so important. It’s also at this point when you should break this up to have variables belonging to the same task grouped together. Such as the following.

...
int  car_mileage, next_oilchange_mileage, oilchange_count;
date next_oilchange_date;
double rent_amount;
date   rent_due_date;
int    rent_total_payments;
...

For booleans use is, did, had, or some other preposition to denote this is a true or false statement. isEligible is a good identifier for an offering and better than eligible. eligible could mean more than just a boolean in certain cases.

For numbers use total, count, square_foot, or something that can define an iterative value.

For strings and characters, try to use something that is neither iterative or prepositional.

Data Structures

Data structures, arrays, dictionaries, sets, are commonly used throughout many languages. Just like variables, they should have descriptive names.

When working with arrays, you will almost always want to use a plural name such as animals = ['dog', 'cat', 'bird', 'fish'] or carMakes = ['audi', 'bmw', 'chevrolet', 'ford', 'mercedes benz', 'nissan', 'toyota'].

The names should be descriptive of what the array contains. It doesn’t make sense when you read a loop that looks like this:

for item in arr:
    do_something_with(item)

Instead, we should be deliberate in how we write this out. Using the animals array above we could rewrite this to show intent on how we want to write this out.

for animal_type in animals:
    do_something_with(animal_type)

Dictionaries and sets should be plural as well when you see animal_names you expect multiple items to belong to the dictionary. By using a descriptive name you could make a decent guess at what might belong to the dictionary: 
animal_names["dog"]

When you use arrays, dictionaries, and sets you should be mindful of how you name them so they look and sound appropriate when you use them later in your code. Poor naming leads to poor readability.

When writing out collections that will contain more than 80 characters worth of items, it is more useful to split each item out on its own line.

names = {'Alice', 'Bob', 'Charlie', 'Doug', 'Edward', 'Frank', 'George', 'Harry', 'Ingrid', 'James'}

// Might look better as
names = {'Alice, 'Bob', 'Charlie, 'Doug', 'Edward', 'Frank',
         'George', Harry', 'Ingrid', 'James'}

// Or even better as
names = {
    'Alice',
    'Bob',
    'Charlie',
    'Doug',
    'Edward',
    'Frank',
    'George',
    'Harry',
    'Ingrid',
    'James'
}

Functions

Naming functions properly is another way to make your code more readable. Functions should be named according to their intent of use. I know in my placeholders I tend to use do_stuff() but that’s because I don’t really care what is going to happen next, after all it is just an example. But when I do care, I use descriptive names.

I used an image of a car above but I really only care about how you start a car. When I get in a car the first thing I do is look for the ignition slot to start the car. When I first use a new library I look for a function that helps me get started with using the library. Sometimes, this is as simple as initializing the class that comes with the library. But what if it’s converting a date to ISO 8601 format (yyyy-MM-ddThh:mm.ssZ)?

string dateToISO8601String(Date date) { ... }
myDateString = dateToISO8601String(myDate);

This shows intent on what the function does. It takes in a date and from the name of the function we could determine that whatever is passed in will result in an ISO 8601 formatted string, useful for most logs.

At this point, we don’t just worry about names, but also whole project structure since functions show up at the root level throughout basic code files.

Some languages allow you to put functions at the root level of your project. In these languages, there usually is one requirement: the function must be defined/stubbed before it can be used.

In C we might do something like this:

#include <stdio.h>
#include <stdlib.h>

int add(int a, int b)
{
    return a + b;
}

int main(int argc, char *argv[])
{
    int n1, n2, sum;
    n1 = atoi(argv[1]);
    n2 = atoi(argv[2]);
    sum = add(n1, n2);
    printf("%d", sum);
}

Or with a stub:

#include <stdio.h>
#include <stdlib.h>

int add(int a, int b);

int main(int argc, char *argv[])
{
    int n1, n2, sum;
    n1 = atoi(argv[1]);
    n2 = atoi(argv[2]);
    sum = add(n1, n2);
    printf("%d", sum);
}

int add(int a, int b)
{
    return a + b;
}

In Python :

import sys

def add(a, b):
    return a + b

n1 = int(sys.argv[1])
n2 = int(sys.argv[2])
sum = add(n1, n2)
print(str(sum))

Now I know I used a and b in the method, but it’s so simple and generic that we don’t really care what a and b are, we just want them to be numeric and add two values together.

You might notice that I put two lines between the import and include lines and the start of the body of the program. I started doing this after I learned Python. According to the PEP 8 standard there should be two lines between the import statements and the rest of your code. It just looks cleaner and helps put some space between the requirements and the logic.

I also put some space between main and add because they are unrelated. Even though main eventually calls the add function, add is responsible for adding two numbers together, and main is in charge of gathering the arguments and displaying the result to the screen.

Finally, I added space between each logical section inside of main. Gathering arguments is a section that I grouped together. Both lines perform almost the same action. I have sum on its own. While we are instantiating a variable, we are actually doing so from the result of a function. This is not the same as above so we add some space to note the difference. Last, print is on its own because it has nothing to do with gathering arguments or retrieving a result from a function.

When you see this in the real world, you might have 40 lines grouped together, after reading the first 3 you create a pattern, you skim down to the last entry and see that it’s the same pattern and you assume that all other 36 lines are exactly the same. You conclude that these 40 lines are all responsible for a similar task.

In your own code this is fine, but I want to warn you, that when using a code base from a lesser known source, I suggest that you browse through all 40 entries to make sure that it does exactly what you suspect that it does. It’s easy to slip something in that might have different results than what you expected.

Summary
