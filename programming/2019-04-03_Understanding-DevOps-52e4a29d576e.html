Understanding DevOps

While I have done quite a few tutorials on Swift, I’ve mostly moved on to do more in the realm of DevOps. Something that I’ve noticed is that most people talk about DevOps like it’s just Kubernetes or just pipelines while others call it a mindset. The first thing I want to say is DevOps is not so much a technology but a role that a department plays within a company.

Before I go out and say what DevOps consists of, let’s cover some general ideas of what it is.

  • Every developer and installation technician has performed DevOps at some point.
  • DevOps is required by every company, even if the title is not clearly defined.
  • DevOps teams work in varying degrees of responsibility, just because a team at Company A does X, does not mean a team at Company B would also do X.

Let’s look at a general IT workflow from a developer standpoint.

Developers:

  • Plan for changes to their code
  • Code new changes for the next feature
  • Build the application changes
  • Test the application’s changes for issues
  • Release the changes into the production environment.

For some companies this release cycle takes months, others may be less; this is normal.

The operations team has a different workflow:

Operations:

  • Deploy new code into the correct environment.
  • Monitor the changes for issues
  • Respond to issues, or Feedback

While it seems that developers have more work on their hands because they have more tasks; the operations team almost always has their hands full responding to tickets from users not understanding the new changes or issues that the users face due to bugs.

I’m sure some developers are saying my code is always perfect, I hate to break it to you, but it takes many years of improvements to make your code near perfect. I’ve seen a 20-year old code base still have bugs. It doesn’t mean bad developers wrote it.

I firmly believe the business pushed the deadlines over code completeness causing the developers to make quality sacrifices along the way. As future updates were released the technical debt rose while new bugs got created as features got added. Sure, they went back and fixed those old bugs, but because new features got released, more recent issues became technical debt.

It’s a vicious cycle that many developers have to face. Agile software development was supposed to fix this by having teams work together. Instead, businesses just used the project management team to relay the business demands to developers and left this to be what agile is today. Not cool.

The idea behind agile software development was to give developers a leg to stand on when working on new features for a project. Developers could talk with other teams and work out a plan that would benefit the business requirements. It was to discuss with the business what could get done in the given timeline, or adjust the deadline so that all the requested features could get implemented correctly.

The operations team has its own set of requirements. Operations are required to understand the changes that got introduced in the release, plan for potential problems, respond quickly to any that occur, and report back to the developers on their findings.

While this may sound easy to the developers reading this, you have to remember that your next release cycle outside of break-fix issues will not happen for another month and the operations team will suffer until you fix those pesky bugs. How are they supposed to monitor all that you do plus the systems they manage if they are always putting out fires?

The DevOps Way

There’s always been a feud between the operations team and developers; both accuse the other of being inept. Until you live in the shoes of both worlds for a while, you’ll keep thinking this way.

I started off working in the help desk of a few companies for longer than I wanted. I then moved up to Systems Administrator and cursed at the developers when I understood enough code to realize they were not handling basic errors, like network requests. Every admin knows the only thing you can count on is for something to go down at the most inopportune time.

If you go to any company and ask one of their Admins, who has worked at the company for longer than a year “What are your common issues”, they will almost certainly come back with a spreadsheet of items about the entire environment.

If you ask the same of an equivalent developer, it will be a spreadsheet of items about only the software environment.

If you ask the same of someone outside of the IT department, they will respond with “Our IT department needs work.” Rarely do I hear anything nice.

Agile was supposed to fix this, but instead what it was really supposed to be was lost along the way but somehow still came about in the name of DevOps.

DevOps is the unification of Operations and Development. A typical DevOps workflow looks like this:

  • Plan for the next release
  • Code new features
  • Build the code for testing
  • Test the new and old code
  • Release the code to the production staging area
  • Deploy the code to production
  • Monitor for issues
  • Get user Feedback
  • Repeat with the feedback gained.

This workflow is essentially taking the roles of Developers and Operations and combining them. It does Not mean that Developers and Operations should become one group; it just means that the act of the two groups working together becomes DevOps.

You may have seen/heard of a DevOps team; this is a specialized team that consists of people with experience in both Administration and Development that work to provide a smooth transition between the two groups.

An example using the more popular Atlassian lineup, the DevOps team would be responsible for setting up BitBucket to hold the code repositories, Bamboo to create and maintain pipelines, and the connections to servers where the code will be deployed.

In this scenario, it is still the developers’ responsibility to write tests for the pipelines and to push their code to the repository, and it is still operations’ responsibility to maintain, troubleshoot, and configure the servers where the code is getting deployed.

Another example, the DevOps team installs and configures Gitlab, utilizes the built-in pipelines, builds the Kubernetes cluster, and sets the integration between Kubernetes and Gitlab.

In this scenario, the developers’ responsibility doesn’t change. However, it is upon the operations team to monitor the Kubernetes cluster for issues, review logs, detect failing pods, and perform remediation tasks on problems that occur. The DevOps team might be involved with writing scripts to manage pods or provide solutions to fixes while the operations team is working with the developers on break-fix changes.

The DevOps team may also be involved with writing ReSTful APIs that can be utilized by developers to ensure proper connections to the databases which are managed by the data management team. This way the environment becomes language agnostic.

By using DevOps, you can strengthen your code base due to feedback from operations. You improve the morale between developers and operations by allowing them to peek into each other’s way of doing business.

DevOps naturally leads you to break things into services, i.e., microservices. It does this because feedback occurs so quickly on the operations side, that developers are more willing to quickly fix the bug while it’s fresh in their mind. The developers make the change, push the code, and within 10 minutes a fixed version is available all because of the work done by DevOps. If you have fixes going in every 10 minutes, you are more inclined to break up the code into smaller pieces so you can fix a bug in another service (or module) while the previous fix is getting deployed.

When you start to feel comfortable about DevOps and microservices, you end up naturally moving towards serverless, where you just have functions sitting out there dormant until called upon to perform some line of work before going back to sleep. These are typically so small that the DevOps team can write them, they can be relatable to new-age stored procedures that are called every once in a while to return a subset of data.

So a question that might come up is “Will DevOps teams automate themselves out of a job?” There is so much work that a DevOps team can do that I would have to say no. A DevOps team responsibility is to take the load off of developers so they can do what they do best: write code. They also provide tools to help free up operations to do what they do best: maintain systems.

You can ask any developer if they like deploying code all day and they will say no, likewise, if you ask any operations person if they like troubleshooting issues all day, they will also tell you no. Does this mean everyone would hate working in a DevOps team? I’m going to tell you no. From my perspective, it is cool stuff. Not only do you make your developers happier by not having to build pipelines, but you also please operations by freeing up their time to focus on responding to issues and maintaining the environment.

Earlier I mentioned every developer and operations person had done DevOps at one point in their career if you have ever tested your code and deployed it to a system that was DevOps. If you have ever mitigated an issue that your customer had or wrote a script to automate a process to keep things running smoothly, that was DevOps.

A good book that I recently read that can provide you more insight on how DevOps plays a role within a company is The Phoenix Project by Gene Kim, Kevin Behr, and George Spafford. It is loosely based on the book The Goal by Eliyahu M. Goldratt and Jeff Cox but with an IT mindset. The Phoenix Project is a fictional thriller, and a very quick read, but provides the basis on what DevOps is all about and how the ideas it provides can influence your workflow.
