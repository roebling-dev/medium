Code Structure and Readability Part 5 — Resources

In the previous article of this series, we learned how to clean up our project structure and maybe break some of our code-base up into libraries.

Code Structure and Readability Part 4 — Project Structure
In the previous article we talked about file structure, today we are moving on to talk about project structure.medium.com

While I didn’t go into depth on how to make a library from your own code there are plenty of tutorials out there that tell you exactly how to do this. This series was only meant to talk about ways that you can clean up your code.

In this final part, I want to share some resources to help you write cleaner code for your language. So this is very likely going to be one of my shortest articles.

I want to mention that ANY of the guidelines posted in any language work well for most languages, as long as the guideline does not interfere with an existing guideline in your language.

I would suggest that you find your language first, or the most similar language to it, and read up on the guidelines, better yet, if your language isn’t listed below search for your language and read it there. Then come back and read some of the others.

Bold Text is the official, and my suggested, documentation.

Swift

Of course, I’m going to start with Swift, but it’s the language I know best.

  • Apple’s Guidelines — https://swift.org/documentation/api-design-guidelines/
  • Ray Wenderlich — https://github.com/raywenderlich/swift-style-guide
  • LinkedIn — https://github.com/linkedin/swift-style-guide

Objective-C (Cocoa)

  • Apple’s Guidelines — 
    https://developer.apple.com/library/archive/documentation/Cocoa/Conceptual/CodingGuidelines/CodingGuidelines.html
  • Apple’s Coding Conventions — 
    https://developer.apple.com/library/archive/documentation/Cocoa/Conceptual/ProgrammingWithObjectiveC/Conventions/Conventions.html
  • Ray Wenderlich — https://github.com/raywenderlich/objective-c-style-guide
  • Github — https://github.com/github/objective-c-style-guide
  • NY Times — https://github.com/NYTimes/objective-c-style-guide

Python

  • PEP 8 Python Code — https://www.python.org/dev/peps/pep-0008/

  • Google — https://google.github.io/styleguide/cppguide.html

Java

  • Oracle (hope I don’t owe them $5) — https://www.oracle.com/technetwork/java/codeconventions-150003.pdf

Ruby

  • RuboCop (community driven) — https://github.com/github/rubocop-github/blob/master/STYLEGUIDE.md

General

  • Thoughbot — https://github.com/thoughtbot/guides
  • Powers of 10 (highly recommended) — https://en.wikipedia.org/wiki/The_Power_of_10:_Rules_for_Developing_Safety-Critical_Code

Summary

While none of these are hard and fast rules, they can greatly increase your productivity in your development cycle and I highly recommend that you at least go over your language.

I hope you’ve enjoyed this series, if you have any questions or felt that I missed anything, let me know.
