I agree with you Swift does provide classes that, but I also believe that when dealing with classes you are also dealing a lot with reference types. If you write something sizable, you almost always end up with spaghetti code in OOP. In POP you are just writing requirements, maybe with default implementations, that can be used as if you were inheriting from a class, even on structs. Swift thrives with value types, that’s why we don’t see much in the Swift documentation about reference types or pointers. Apple has us opt for structs over classes so we can use protocols rather than superclasses. Apple doesn’t mention it directly in their documentation but if you add up all of their best practices for Swift development, they all point to using protocols. This makes Swift more protocol-oriented than object-oriented.
