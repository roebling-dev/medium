Failable initializers are at a higher scope than optional properties. While I could have an optional property in a class that is guaranteed to be there such as a Person class that contains non-optional property for firstName, but optional properties for middleName, and lastName (think Prince).

When I call the default initializer I will always have a person’s firstName even if it is blank, but middle and last name would be optional. We’d still have a Person created if we used Person(firstName: “Bob”, middleName: nil, lastName: “Roebling”). It wouldn’t fail because Person(firstName: 1, middleName: 2, lastName: 3) would still be created because Int can be casted to a String.

With a failable initializer we are saying if the class initialization might fail based on the data that was passed in, fail the entire initialization and return nil. If we had a variable named myPerson that was initialized with nil for the firstName property, then we would want to return nil to denote that the class could not be created at all.

In a nutshell, optional properties are used when you plan to return an instance of a class every time the initializer is called. A failable initializer is used when the entire instance doesn’t need to exist. According to Apple’s docs on failable initializers, they would be used when initialization depends on a resource such as data from a JSON file.
