In scenarios where you don’t care if the image gets downloaded or not, it may not be an issue. But if the image download does matter and your user is on a slow connection or loses connection, you have to start the download from the beginning. With URLSession you can handle this and at least store the partial download on disk and resume it when the connection is better. It may not be an issue for small images that are a few KB in size but what if the user is downloading a large 4K image? The biggest misconception is that users will always have a great connection when many parts of the world still have connections comparable to dial-up. Especially when you think about spotty signal on mobile devices.

As an example an app that I’ve been working on contains a 1.2 MB JSON file. On normal connections the user might not realize something was downloaded. On bad connections this could take a few minutes. Even if I push it to a background queue to free up the UI, what if my user is waiting on that data to perform some action? What if the user gets tired of waiting and backgrounds the app? If I’m using data:contentsOf, that download may not finish and cancel out, even if it was 80% downloaded. With URLSession you can either continue to download it in the background or pause the download, save the partial data to disk, then resume it when your app resumes.
